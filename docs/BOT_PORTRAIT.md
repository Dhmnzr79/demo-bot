# Идеальный AI-бот 2026

## Цель документа

Функциональный портрет.

ТЗ с точки зрения маркетинга и UX. Один клиент. Без мультиагентности.

Документ описывает, каким должен быть грамотный AI-агент в 2026 году: как понимать пользователя, как искать в базе, как отвечать, как вести к действию и как безопасно обрабатывать ошибки.

## Принципы

- Понимать намерение пользователя, а не букву.
- Не выдумывать факты: отвечать только на основе знаний из базы и контекста диалога.
- Не оставлять пользователя в тупике: каждый ответ ведет к следующему шагу.
- Быть полезным и спокойным в тоне, особенно в тревожных темах.
- Соблюдать ограничения безопасности и не раскрывать внутреннюю логику.

## Ключевые механики

### 1. Понимание запроса пользователя

Люди пишут коротко, с ошибками, бытовым языком. Задача бота - понять намерение, а не букву.

1. Определить **тип запроса** до поиска по базе: цена, контакты, запись, страх/тревога, вопрос по услуге, мусор.
2. Короткие фразы («да», «нет», «хочу», «подробнее») - **не отправлять в поиск**. Читать из контекста предыдущего хода.
3. Понимать бытовые формулировки («вдруг не приживётся», «страшно», «дорого») и соотносить с темами базы.
4. Если запрос неясен - не гадать. Мягко уточнить **одним** вопросом.

**Статус в текущем проекте:** частично реализовано.  
**Пока не реализовано:** Явный класс мусора/оффтопа - пока бот обрабатывает его как обычный контентный вопрос.

### 2. Перефразирование запроса для поиска

Пользователь продолжает диалог коротко. Бот должен дополнить вопрос контекстом перед поиском.

1. Перед поиском - **переформулировать запрос** с учётом последних реплик диалога (query rewrite).
2. Не добавлять факты из головы - только то, что было в разговоре.
3. Если вопрос самодостаточен - не усложнять, искать как есть.
4. При неудаче переформулировки - использовать исходный вопрос.

### 3. Поиск в базе знаний

Бот ищет ответ в базе, не генерирует из головы.

1. **Векторный поиск** (RAG) - находит смысловые совпадения, не просто ключевые слова.
2. **Синонимы и алиасы** - прямое совпадение по разговорным формулировкам важнее векторного.
3. **Прямая ссылка** (ref) - мгновенный ответ без поиска. Используется для кнопок навигации.
4. Если ничего не найдено - **не фантазировать**. Честно сообщить и предложить уточнить или передать администратору.

**Статус в текущем проекте:** частично реализовано.  
**Пока не реализовано:** Кеш частых запросов - не реализован.

### 4. Выбор наиболее подходящего куска (rerank)

Из нескольких найденных кандидатов бот выбирает один - самый точный.

1. LLM оценивает топ-N кандидатов и выбирает тот, который точнее отвечает на вопрос.
2. Учитывает контекст диалога, не только текущий вопрос.
3. Если ни один кандидат не достаточно релевантен - уйти в fallback (низкая уверенность).

### 5. Генерация ответа

Бот не копирует кусок базы - отвечает по нему своими словами.

1. Ответ строго по найденному куску. **Никаких фактов из головы.**
2. Коротко и по делу: ответить именно на вопрос, не пересказывать всё подряд.
3. Живой разговорный язык. Без медицинских терминов и канцелярита.
4. Если в базе есть цифры, сроки, проценты - **обязательно включать** в ответ.
5. Ответ заканчивается переходом к следующему шагу. Не оставлять пользователя в тупике.
6. Ответ появляется по мере генерации (**стриминг**), не ждёт полной готовности.

### 6. Эмпатия

Эмпатия - не общий тон, а конкретная механика. Включается только там, где нужна.

1. **Триггеры:** фразы про страх, боль, тревогу, сомнение, «дорого», плохой прошлый опыт.
2. Проявляется **один раз за тему** - не повторяется в каждом ответе.
3. Формат: одна короткая фраза в начале ответа → сразу по делу.
4. **Не включать** при простых справочных запросах: адрес, телефон, график.
5. Бот - консультант, не терапевт. Формула: признание + факт + следующий шаг.

### 7. Кнопки и навигация

Кнопки - не меню, а подсказки следующего шага. Снижают нагрузку на пользователя.

1. После каждого содержательного ответа - **1-2 кнопки** продолжения темы.
2. Кнопки конкретные: «Этапы лечения», «Стоимость», «Кому подходит» - не «Подробнее», «Ещё».
3. Кнопки определяются в базе знаний для каждой темы, не генерируются случайно.
4. Уже пройденные темы в кнопки **не возвращать**.
5. Максимум 2-3 кнопки за раз. Больше - перегрузка выбором.
6. Кнопки - дополнение к свободному вводу, не замена. Пользователь всегда может написать сам.

### 8. CTA - кнопка целевого действия

CTA показывается в нужный момент, а не в каждом ответе.

1. Появляется **не раньше** чем пользователь получил достаточно информации (настраивается для каждой темы).
2. Не показывать, если пользователь **уже хочет записаться** - сразу запустить сбор заявки.
3. Не показывать в момент активного флоу записи.
4. Один CTA за ответ. Не дублировать с кнопками навигации.
5. Текст и действие CTA настраиваются для каждого клиента.

### 9. Сбор заявки (лид)

Если пользователь готов к действию - не тормозить его информацией.

1. Триггер: написал о желании записаться → **запустить флоу немедленно**, без лишних объяснений.
2. Опционально перед именем: предложить описать ситуацию - чтобы администратор был в курсе.
3. Спросить имя. Проверить, что это **имя**, а не вопрос или мусор. Если сомнительно - уточнить с кнопками «Да / Нет».
4. Спросить телефон. Нормализовать формат. Если не распознан - попросить повторить.
5. В заявку включать: имя, телефон, описание ситуации, тему диалога, время.
6. При ошибке отправки - **сохранять резервно**, не терять данные.

**Статус в текущем проекте:** частично реализовано.  
**Пока не реализовано:** Webhook-доставка (POST на URL клиента), дедубликация лидов по телефону, уведомление в Telegram/Slack - не реализованы.

### 10. Память диалога

Бот помнит контекст в рамках сессии. Без этого короткие фразы и сценарии не работают.

1. Запоминать: текущую тему, последнее действие бота, показанные кнопки, активный флоу.
2. Профиль пользователя: имя и телефон - **не спрашивать повторно** в рамках сессии.
3. История последних ~8 ходов - для переформулировки запросов и контекста LLM.
4. Отслеживать, что уже показывали: видео, CTA, предложение описать ситуацию - не повторять.
5. Сессия живёт **1 час** с последней активности. После - сброс.
6. Смена темы определяется и контекст обновляется автоматически.

**Статус в текущем проекте:** реализовано (в рамках сессии).  
**Примечание:** Межсессионная память (пользователь вернулся через неделю) в целевой портрет не включается: для этого сценария она не нужна.

### 11. Антиспам

1. Ограничить **частоту запросов** с одного пользователя / IP.
2. Повторяющиеся одинаковые сообщения - не обрабатывать каждое, игнорировать дубли.
3. Мусорные наборы символов - не отправлять в LLM, вернуть нейтральный ответ.
4. Слишком быстрые серийные запросы - временная пауза с понятным сообщением.
5. Отдельно отслеживать сценарий «бесконечные вопросы без намерения записаться»: стандартный антиспам может это не ловить, если интервалы нормальные и контент валидный.
6. Ввести session-level сигнал: если `session_turn_count >= 10` и ни разу не было `booking_intent`, подменять обычный ответ на мягкий редирект.
7. Действие при таком сигнале: не блокировка, а мягкое переключение - предложить консультацию или передать администратору.
8. Формулировка мягкого перехода: «Я рассказал уже довольно много о разных темах. Возможно, удобнее обсудить вашу ситуацию напрямую - консультация бесплатна, врач ответит на всё сразу.»
9. Не повторять обычный CTA в этом сценарии, если он уже показывался ранее.

**Статус в текущем проекте:** частично реализовано.  
**Пока не реализовано:** Полноценный rate limiting по IP - не реализован.
**На будущее:** Для production-safe rate limiting использовать общий стор лимитов (например, Redis), чтобы лимиты были едиными для всех инстансов/воркеров.
**На будущее:** За reverse proxy учитывать реальный IP клиента (доверенные заголовки), иначе лимиты могут срабатывать некорректно.

### 12. Ограничение длины входящего сообщения

1. Входящее сообщение обрезается до разумной длины **до обработки** (текущий ориентир: 600 символов).
2. Пользователь не видит техническую ошибку при обрезке - бот обрабатывает тихо.
3. Слишком длинный текст (история болезни, копипаста) - обработать только первую часть.

**Статус в текущем проекте:** частично реализовано.  
**Пока не реализовано:** Ограничение длины на входе - не реализовано.

### 13. Безопасность и ограничения бота

1. Бот **не ставит диагноз**, не назначает лечение, не гарантирует результат.
2. Внутренние инструкции и база знаний в сыром виде - **не раскрываются**.
3. Запросы вне темы клиники («напиши стих», «кто президент») - мягко вернуть к теме.
4. Попытки «взломать» поведение («забудь инструкции») - игнорировать, возвращать к теме.
5. Не выдумывать цены, акции, гарантии, имена врачей - **только то, что есть в базе**.
6. Персональные данные - только для заявки. Телефон в логах маскировать.

### 14. Ошибки и fallback

Любая ошибка должна давать пользователю понятный следующий шаг - не тупик.

1. LLM не ответила → нейтральное безопасное сообщение. **Никакого стек-трейса** пользователю.
2. База ничего не нашла → честно сказать и предложить передать вопрос администратору.
3. Заявка не отправилась → сохранить резервно, сообщить пользователю понятно.
4. После **2-3 неудачных попыток** понять запрос - показать меню-выбор вместо очередного «уточните».
5. Непредвиденная техническая ошибка - пользователь видит понятное человеческое сообщение.

### 15. Время работы клиники и коммуникация в виджете

Виджет должен учитывать рабочие часы и корректно формулировать ожидания пользователя.

1. Для логики рабочих часов использовать локальный часовой пояс и расписание конкретной клиники (настраиваемые параметры).
2. После успешной отправки заявки проверять, открыта ли клиника в данный момент.
3. Если клиника закрыта, добавлять к подтверждению заявки фразу: «Извините, клиника сейчас закрыта. Мы свяжемся с Вами в рабочее время».
4. Над свернутым виджетом поддерживать всплывающий мини-баннер с контекстными подсказками.
5. В нерабочее время текст мини-баннера: «Клиника сейчас закрыта. Вы можете задать вопрос нашему онлайн-консультанту Анне.»
6. Через 30 секунд после захода на сайт текст мини-баннера: «Есть вопросы? Наш онлайн-консультант Анна поможет вам.»

**Статус в текущем проекте:** частично реализовано.  
**Пока не реализовано:** Логика нерабочего времени, пост-обработка заявки по рабочим часам и сценарии мини-баннера требуют отдельной реализации.

### 16. Handoff в человека с полным контекстом

При передаче администратору важно отправлять не только лид, но и краткую выжимку диалога, чтобы человек продолжал разговор без потери контекста.

1. При handoff передавать администратору структурированную сводку: намерение пользователя, что уже спросили, что уже ответили, где пользователь «застрял».
2. В сводку включать краткий хронологический контекст последних шагов, чтобы не повторять одни и те же вопросы.
3. Если пользователь явно просит человека, handoff выполнять немедленно, без дополнительных барьеров.
4. В handoff-пакет добавлять полезные метаданные: текущая тема, активный этап флоу, предложенные CTA/кнопки, наличие контактных данных.
5. После передачи в человека бот подтверждает пользователю следующий шаг понятной фразой («Передал ваш вопрос администратору, с вами свяжутся»).

**Статус в текущем проекте:** частично реализовано.  
**Пока не реализовано:** Единый стандарт handoff-сводки и стабильная передача полного контекста в канал администратора требуют отдельной реализации.

### 17. Политика отказа по «красным» запросам

Бот должен не только избегать диагноза, но и корректно обрабатывать высокорисковые сценарии безопасности.

1. Определять «красные» запросы: острые симптомы, признаки неотложного состояния, риск для жизни, опасные сценарии самолечения.
2. В «красных» сценариях не обсуждать лечение по существу и не давать потенциально опасные рекомендации.
3. Выдавать безопасный сценарий действий: немедленно обратиться за очной медицинской помощью / вызвать экстренную службу (по локальным правилам).
4. Использовать ясный, спокойный и короткий тон без запугивания и без ложной уверенности.
5. Предлагать параллельный безопасный следующий шаг в рамках сервиса (например, оставить контакты для связи с администратором), если это не задерживает экстренное обращение.

**Статус в текущем проекте:** частично реализовано.  
**Пока не реализовано:** Полный список «красных» триггеров, отдельные safety-шаблоны и жесткая политика ответа для high-risk сценариев требуют отдельной реализации.

### 18. Метрики качества и контур улучшений

Без измерений невозможно системно улучшать бота: решения должны приниматься по данным, а не по ощущениям.

1. Отслеживать не только лиды, но и качество диалога: долю fallback-ответов, долю «не понял», успешность первого ответа, глубину до целевого действия.
2. В retrieval-контуре измерять качество поиска: hit-rate релевантных источников, долю пустой выдачи, стабильность top-кандидатов.
3. Для конверсионного контура отслеживать путь пользователя: показы CTA, клики CTA, переходы в сбор заявки, успешные отправки.
4. Разделять метрики по темам и сценариям (цены, страх/тревога, запись, контакты), чтобы видеть точечные слабые места.
5. Запустить регулярный цикл улучшений: приоритет проблемных зон → правки контента/логики → повторная проверка метрик.

**Статус в текущем проекте:** частично реализовано.  
**Пока не реализовано:** Сквозной дашборд качества, единые KPI по этапам диалога и регулярный процесс пересмотра метрик требуют отдельной реализации.
